# Section 6 — Engineering Team

**Paper patterns: Parallelization + Orchestrator-subagents**

From [Building Effective Agents](https://www.anthropic.com/engineering/building-effective-agents): parallelization applies when a task can be broken into independent subtasks that run simultaneously, or when multiple independent agents check each other's work. Combined with orchestrator-subagents, this produces pipelines where some work fans out to specialised agents working in parallel while overall direction stays coherent.

---

## What this demonstrates

Section 4 introduced a manager orchestrating sequential workers. Section 6 shows the next step: a DAG (directed acyclic graph) where tasks fan out rather than chain linearly. The backend engineer reads the design; the frontend engineer and test engineer both read the backend code, but independently of each other. They never share context directly. This mirrors the information flow of a real engineering team: the test engineer reads the implementation, not the design doc.

**Context chains model information flow explicitly.** `code_task` has `context: [design_task]`. Both `frontend_task` and `test_task` have `context: [code_task]` — they receive the backend implementation, not the design. The engineering lead's thinking never reaches the test engineer directly; it flows through the code that implements it. This is intentional: passing less context to each agent forces the agent to work with what it actually needs, and reduces the noise of information that is technically available but not relevant to the agent's specific task.

**Specialisation improves output quality.** A single agent asked to design, implement, test, and build a UI produces shallow work across all four areas. Separate agents with focused backstories produce a design document that looks like a design document, implementation code that looks like production code, and tests that cover edge cases — because each agent is primed to think like that specialist. The engineering lead received a minimal requirements spec and produced a custom exception hierarchy, UUID-tagged transactions, and historical time-travel queries. None of that was in the spec; it came from the agent's engineering-lead identity.

**Mixed execution modes match real workflows.** The backend engineer and test engineer have `allow_code_execution=True` — they verify their work by running it. The engineering lead and frontend engineer do not run code; their outputs are a design doc and a Gradio app, respectively. Enabling code execution only where it provides real verification keeps the pipeline focused and avoids paying for capability that has no useful feedback loop.

---

## Key files

- `config/agents.yaml` — four agents: `engineering_lead` (gpt-4o), `backend_engineer`, `frontend_engineer`, `test_engineer` (gpt-4o-mini)
- `config/tasks.yaml` — four tasks with explicit `context:` creating a DAG: design → code → {frontend, test}
- `crew.py` — `backend_engineer` and `test_engineer` have `allow_code_execution=True`; output files under `output/engineering_team/`

## What to observe

**Run it:**
```bash
uv run python main.py engineering_team
```

After the run, check all four output files exist: `*_design.md`, `*.py`, `app.py`, `test_*.py`. If `accounts.py` is missing, the pipeline continued (downstream agents got the code via context) but the file was not written — a sign the backend engineer narrated rather than outputting raw Python.

```bash
uv run python output/engineering_team/app.py
```

Running this launches the Gradio UI in a browser — a complete, functional interface generated by an agent that has never run the code it is wrapping.

**Experiments to try:**

Compare the design doc to the original requirements in `main.py`. Note every decision the engineering lead made that was not asked for — exception hierarchy, UUIDs, historical queries, serialisation. These are the places where model capability shows most clearly as the agent applies engineering judgement rather than transcribing the spec.

Remove `context: [design_task]` from `code_task` in `tasks.yaml` and rerun. Does the backend engineer still produce a coherent implementation without seeing the design? Observe whether removing upstream context improves or degrades the output — the answer is not obvious.

Add a fifth agent — a `code_reviewer` — that reads both `design_task` and `code_task` via context and writes a markdown review noting where the implementation diverges from the design.
