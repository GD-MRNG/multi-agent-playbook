# Phase 5 — Engineering Team: Specialisation, Branching Context, and Mixed Execution

## Why This Conversation Is Happening

Phases 1 through 4 explored a single expanding idea: what a single agent can do when you give it better tools, better structure, and the ability to run code. Phase 5 is a different kind of question. The engineering team crew is not about what one agent can do — it is about what happens when you stop asking one agent to do everything and instead compose a team of specialists whose information flow mirrors the real dependencies between their roles.

The crew takes a natural-language requirements spec and produces four artifacts: a design document, a Python module, a Gradio UI, and a unit test suite. Each is written by a different agent whose identity was shaped to produce exactly that artifact. The engineering lead thinks like a senior designer — it produced a production-grade exception hierarchy, a typed `Transaction` dataclass with UUIDs and timestamps, and historical time-travel query methods, none of which appeared in the spec. The backend engineer thinks like an implementer — it followed the design carefully and produced clean, executable code. The test engineer thinks like a QA engineer — it wrote tests against the actual API the backend engineer produced, not the one the requirements described. The frontend engineer wrapped everything in a Gradio UI using `gr.State` to hold the account object across interactions, a pattern that would not have appeared from a generic agent.

Three things in this phase have no equivalent in the earlier phases. First, context stops being a linear chain and becomes a directed graph: two tasks — frontend and tests — both read from the same upstream task independently, not from each other. Second, the decision of which agents get code execution capability and which do not becomes meaningful: the backend and test engineers need to verify their output by running it; the engineering lead and frontend engineer produce artifacts intended for a human to run later. Third, because the engineering lead exceeded the spec — producing a richer API than was asked for — the downstream agents adapted to what they actually received rather than what was planned. That adaptability is a property of the context mechanism, and understanding it changes how you reason about pipeline consistency.

## What You Need To Know First

**What a directed acyclic graph (DAG) is.** A DAG is a set of nodes connected by directed edges, with no cycles. "Directed" means each edge has a direction (A → B means A feeds B, not the reverse). "Acyclic" means you cannot follow edges and return to where you started. Task dependencies in a multi-agent pipeline form a DAG: each task points to the tasks whose output it needs, and no task can depend on a task that depends on it. In this crew, `code_task` has two outbound edges — to `frontend_task` and to `test_task` — which makes the graph branch rather than stay linear.

**What Gradio is.** Gradio is a Python library for building browser-based UIs for machine learning and data applications. A minimal Gradio app is roughly twenty lines of Python: a `gr.Blocks()` context, some input and output components, and event listeners wired to Python functions. The resulting UI runs in a browser on localhost. It is the standard prototyping layer for AI applications that need a human-facing interface without the overhead of a full web framework. The frontend engineer in this crew writes that file; it does not run it.

**What unit tests are and why they must run against code.** Unit tests are short programs that call specific functions or methods with known inputs and assert the outputs match expected values. They verify that code behaves correctly across a range of cases, including edge cases and error conditions. The critical point for this crew is that unit tests are only meaningful when run against the actual implementation — a test that cannot execute is documentation, not verification. The test engineer has `allow_code_execution=True` specifically so it can run the test suite and see real pass/fail results rather than predicting them.

## The Key Ideas, Connected

**Specialisation works because each agent's identity constrains its output style.** The engineering lead's goal instructs it to "describe the function and method signatures" — so it produces a design document. The backend engineer's backstory says "you follow design instructions carefully and produce 1 python module" — so it produces a self-contained module. The test engineer's backstory establishes it as a QA engineer — so it writes edge-case tests rather than happy-path-only coverage. These are not just role labels; they are the same mechanism from Phase 1 — backstory as constrained system prompt — applied at scale across four distinct professional identities. An agent whose identity matches its task produces output that looks like the work of someone who does that job.

**Context in this crew is a directed graph, not a pipeline.** In Phase 1 and 2, context was linear: task A fed task B, which fed task C. Here, `code_task` feeds two tasks independently: `frontend_task` needs the implementation to know what to import; `test_task` needs the implementation to know what to test. Neither the frontend engineer nor the test engineer reads the design document directly — they read the code. This reflects the real dependency: tests verify the implementation, not the spec; a UI wraps the code, not the design. The `context:` field in `tasks.yaml` is where you encode this graph. The test engineer's task lists `context: [code_task]`, not `context: [design_task, code_task]`. The design reaches the test engineer only if it reaches the code, which it does — but passing it directly would be noise, not signal.

**Which agents get code execution is a design decision, not a default.** The backend engineer runs the module to verify it imports and executes without error. The test engineer runs the test suite to confirm tests pass. Both of these are verification steps — the agent produces output, then checks whether the output is correct by running it. The engineering lead and frontend engineer do not run their output: the design document is markdown, which does not execute; the Gradio app needs a browser to render, which the Docker sandbox cannot provide. Giving the frontend engineer `allow_code_execution=True` would let it run Python but not verify the UI — the execution capability would add cost without adding signal. Matching execution capability to tasks where it provides real feedback, and withholding it where it does not, is the right way to configure a mixed crew.

**Raw code output requires explicit suppression of markdown formatting in every task that writes a `.py` file.** Language models default to wrapping code in markdown fences — ` ```python ` at the start, ` ``` ` at the end. This is correct behaviour in a chat interface, where the output is read by humans. It is incorrect for a task whose `output_file` is a `.py` file that will be imported or executed. Both `code_task` and `frontend_task` and `test_task` include the same instruction in `expected_output`: "Output ONLY raw Python code — no markdown formatting, no code fences, no backticks." This must be in every code-writing task independently; it is not inherited from the agent backstory. One missing instruction produces a file that fails on import with a syntax error on the first line.

**Downstream agents adapt to what they actually received, not what was specified.** The requirements said nothing about exception hierarchies, `Transaction` dataclasses, or historical time-travel queries. The engineering lead added them. The backend engineer implemented them. The test engineer then wrote tests for `InsufficientFundsError`, `InsufficientSharesError`, and `get_holdings(at_time=...)` — because those are in the code it received via context, even though they were not in the original spec. The frontend engineer used `gr.State` to hold the account object, because the implementation warranted it. This is the pipeline staying internally consistent through agent-to-agent handoffs: each downstream agent reads what it actually received and adapts, rather than working from a fixed expectation. The consequence is that if the backend engineer produces a different API than the design specified, the tests and UI will match the actual implementation — not the planned one. That is usually the right outcome.

## Handles and Anchors

The `context:` field is a CC list, not a forwarding rule. When you CC someone on an email thread, you are making a deliberate decision: this person needs this information to do their job. You do not CC everyone on everything — you think about who needs what. `context: [code_task]` on the test task means: the test engineer is CCed on the code. Not the design, not the frontend — the code. The CC list documents the real information dependency.

Mixed execution modes mirror real team composition. A senior backend engineer runs the module before declaring it done — not to prove they wrote it, but to confirm it actually works. A QA engineer runs the test suite to see whether tests pass or fail, not to guess. A technical writer who produces a design document does not need to execute it; a UI designer who produces a wireframe does not need to run it in a browser. Matching execution capability to roles where running produces useful feedback is the same judgment call a team lead makes when deciding who gets access to the deployment environment.

The LLM extrapolating beyond the spec is like hiring a senior engineer rather than a contractor. A contractor delivers exactly what was specified. A senior engineer reads the requirements, identifies the gaps, and adds the things that would have been painful to retrofit: a custom exception hierarchy so callers can handle errors precisely; UUIDs on transactions so records are auditable; historical queries so the portfolio value can be checked at any past point in time. None of this was in the requirements. All of it was in the engineering lead's output. The downstream agents worked with what they received, not with what was asked for. That alignment — between what was produced and what downstream agents consumed — is the pipeline behaving as intended.

## What This Changes When You Build

An engineer who understands context as a directed graph will draw the dependency graph before writing `tasks.yaml`. Which task needs which prior output? Who should be CCed on the code? Who should see only the design? The graph answers those questions, and `context:` fields are how you encode the graph. A context chain where everyone gets everything is not a design — it is an absence of design.

An engineer who understands specialisation will write agent backstories that match the output style of the role, not just the role label. "You're a QA engineer" is a label. "You're a seasoned QA engineer who writes thorough unit tests for Python code" is a backstory — it shapes the agent toward edge-case coverage and test completeness rather than superficial happy-path verification. The richness of the backstory correlates with the depth of the output.

An engineer who understands mixed execution modes will ask, for each agent: does running the output produce useful feedback that the agent can act on? If yes, enable execution. If no — because the output requires a runtime environment the sandbox cannot provide, or because the output is not executable — withhold it. The cost of execution capability is real (Docker startup, token burn on execution loops); it should be justified by signal.

An engineer who understands raw code output requirements will add the markdown-suppression instruction to every task that writes a `.py` file, and will verify the output files after each run. A file that starts with ` ```python ` fails silently in the pipeline — downstream agents still receive the content via context, so the crew finishes, but the file on disk cannot be executed. Verification means opening the file, not just confirming it exists.

An engineer who has seen downstream agents adapt to richer-than-specified upstream output will stop treating the spec as a guarantee of what the pipeline produces. The spec is the input. What the pipeline produces is the output of the agent that read the spec, which may be better, different, or more elaborate. The test engineer tests what was built, not what was planned. That alignment is a feature — it means the tests are honest — but it requires the engineer to read the actual output files rather than assuming they match the spec.
